# CLAUDE.md — 태백가덕산 풍력 발전량 예측 대회

이 문서는 Claude Code가 이 프로젝트에서 작업할 때 항상 지켜야 할 규칙과 컨텍스트를 정의한다.

## 1. 프로젝트 개요

- **대회**: DACON 제3회 풍력발전량 예측 AI 경진대회 (대회 기간 2026-07-06 ~ 08-14)
- **과제**: 기상 예보(LDAPS, GFS)와 SCADA 실측 데이터를 활용해 2025년 1년치(8,760시간) KPX 그룹별 시간당 풍력 발전량(kWh)을 예측한다.
- **평가 구조**:
  - 리더보드: 평가 데이터 중 사전 샘플링된 **40%가 Public**, 나머지 **60%가 Private**. 1차 평가는 **Private Score 100%**.
  - 제출: **1일 최대 5회**, 마감 2026-08-14(금) 10:00. 순위가 가장 높은 파일이 자동 선택되나 직접 지정도 가능.
  - 2차 평가: Private 상위 30팀이 산출물 제출(~08-17 10:00) → 검증 통과 상위 20팀이 오프라인 발표평가 → **최종 점수 = Private 50% + 발표 50%**로 상위 10팀 수상.
  - 따라서 리더보드 점수만큼 **재현 가능한 코드와 발표 자료의 품질이 수상을 좌우한다** (12번 규칙 참조).
- **대상 발전단지**: 태백가덕산 풍력 (강원 태백, 약 37.28°N / 128.95°E, 산악 지형)
- **예측 대상 3개 그룹**:

| 그룹 | 구성 | 설비용량 | 1시간 환산 |
|---|---|---:|---:|
| `kpx_group_1` | VESTAS V126 1~6호기 (3.6MW×6) | 21.6 MW | 21,600 kWh |
| `kpx_group_2` | VESTAS V126 7~12호기 (3.6MW×6) | 21.6 MW | 21,600 kWh |
| `kpx_group_3` | UNISON U136 1~5호기 (4.2MW×5) | 21.0 MW | 21,000 kWh |

- 모든 터빈 **허브 높이 117m**, 로터 직경 V126=126m / U136=136m. (`info.xlsx` 참조)
- 그룹 1·2는 같은 기종·같은 단지라 발전 패턴이 매우 유사할 것. 그룹 3은 기종이 다르고 라벨이 2023년부터만 존재.

- 기상 데이터는 `forecast_kst_dtm` 하나당 LDAPS 16행 / GFS 9행 (grid별). 반드시 **격자를 컬럼으로 pivot**하거나 집계해서 시간당 1행 구조로 만든 뒤 라벨과 조인한다.
- `kst_dtm` / `forecast_kst_dtm`은 집계 구간의 **종료 시각**이다 (01:00 값 = 00:00~01:00 발전량).

## 3. Git 사용 규칙

**커밋은 사용자가 직접 한다. Claude는 git 상태를 바꾸는 어떤 명령도 실행하지 않는다.**

- **금지 명령 (절대 실행 금지)**: `git add`, `git commit`, `git push`, `git pull`, `git merge`, `git rebase`, `git reset`, `git restore`, `git checkout`, `git stash`, `git branch -d/-D`, `git tag`, `git clean` 등 저장소 상태·워킹트리를 변경하는 모든 git 명령. 사용자가 명시적으로 요청해도 "커밋은 사용자 담당"임을 상기시키고 명령어만 알려준다.
- **허용 명령 (읽기 전용)**: `git status`, `git diff`, `git log`, `git show`, `git rev-parse` 등 조회성 명령은 자유롭게 사용해 컨텍스트를 파악해도 된다.
- **커밋 시점 제안**: 의미 있는 작업 단위가 끝나면 (예: 셀 묶음 완성, Phase 완료, 제출 파일 생성) 사용자에게 "지금 커밋하기 좋은 시점입니다"라고 알리고, 변경된 파일 목록과 함께 **커밋 메시지 추천안**을 제시한다. 예: `[phase1] EDA: 예보-실측 풍속 상관 분석 추가`, `[submit] exp012: 허브고도 외삽 피처, val total_score 0.xxxx`. 실행은 하지 않는다.
- **.git/ 폴더**: 직접 열거나 수정하지 않는다.
- **.gitignore**: 수정이 필요하다고 판단되면 (예: `data/processed/`가 추적되고 있음, 대용량 parquet이 커밋될 위험) 변경안을 사용자에게 제안하고, 승인받은 경우에만 `.gitignore` 파일을 편집한다. 권장 무시 목록: `venv/`, `data/`(원본 대용량 CSV), `data/processed/`, `__pycache__/`, `.ipynb_checkpoints/`. 반대로 `노트북, src/, reports/, experiments/, submissions/`는 추적 대상으로 두는 것을 권장한다 (실험 이력이 히스토리로 남아야 함).
- **대용량 경고**: 새로 생성한 파일이 큰 경우 (수십 MB 이상) 사용자에게 "이 파일은 커밋하지 말거나 .gitignore에 추가하세요"라고 알린다.
- **실험 재현성**: 제출 파일을 만들 때 `git rev-parse --short HEAD`(읽기 전용)로 현재 커밋 해시를 조회해 실험 로그에 기록한다. 워킹트리에 커밋 안 된 변경이 있으면(`git status`로 확인) 그 사실도 로그에 남기고 사용자에게 커밋을 권한다.

## 4. 데이터 누수(leakage) 절대 규칙 — 대회 공식 규칙, 위반 시 실격 사유

**기본 원칙 (예측기준시점)**: 각 예측값에는 그 예측의 **예측기준시점 이전에 생성·공개·확정되어 실제로 활용 가능했던 정보만** 사용할 수 있다. 사용 가능 여부는 데이터가 가리키는 대상 시각이 아니라 **데이터가 실제로 활용 가능해진 시각** 기준이다. 이 대회에서는 전일 13:00(KST)에 공개되는 day-ahead 예보로 다음날을 예측하는 구조이며, 제공된 `data_available_kst_dtm`이 그 기준이다.

- **기상 예보**: 제공된 예보 컬럼(`data_available_kst_dtm` ≤ 예측기준시점)만 쓰면 자동으로 만족된다. 외부 예보 데이터를 추가할 경우에도 "예보 생성·공개 시각" 기준으로 판단한다.
- **실측 데이터**: 예측기준시점 이전에 이미 관측·확정된 과거 자료만 사용 가능. **사후 보정자료·재분석자료(ERA5 등)는 평가 기간에 대해 사용 금지.**
- **테스트 기간(2025년)에는 SCADA가 없다.** SCADA는 test 시점 feature로 직접 쓸 수 없다. 허용되는 용법: (a) 학습 시 보조 타깃/사전학습, (b) 예보 풍속 → 실측 풍속 보정 모델 학습, (c) 터빈별 파워커브 추정 — 모두 학습 기간 데이터로만 수행.
- **파생 변수 규칙 (공식 규칙 명시)**: lag, rolling, 보간, 집계, 통계값 등 어떤 파생 변수에도 예측기준시점 이후의 정보가 포함되어선 안 된다. 실전 체크: 라벨 기반 lag/rolling 피처는 test에서 존재하지 않으므로 사용 불가(또는 예보만으로 계산 가능한 형태여야 함), 보간 시 미래 값 참조 금지.
- **평가 데이터셋(test)은 추론 전용이다 (공식 유의사항)**: test 기간 데이터로 어떤 학습도 하지 않는다. 구체적으로 — 스케일러·인코더 등 **전처리 통계는 반드시 train에서만 fit**하고 test에는 transform만 적용, test 피처를 이용한 의사 라벨링·test-time adaptation·test 분포 기반 보정 금지.
- **금지 정보**: 평가 구간의 실제 발전량, 실측 운영 데이터, 비공개 운영 데이터, 정답에 준하는 정보(예: KPX 공개 발전 실적을 수집해 2025년 정답을 유추)의 직·간접 활용은 **즉시 실격** 사유다.
- 검증은 반드시 **시간 기반 분할**로 한다 (예: 2022~2023 학습 / 2024 검증). 랜덤 K-Fold 금지. 최종 제출 전 2024년 전체를 홀드아웃으로 대회 산식 그대로 평가한다.
- **소명 의무**: 전처리·학습·추론 전 과정이 위 규칙을 준수했음을 2차 평가에서 설명할 수 있어야 한다. 따라서 누수 관련 설계 판단(예: "스케일러는 train만 fit — 규칙 4")은 코드 주석과 `reports/` 문서에 근거로 남긴다. 소명이 불충분해도 평가 제외될 수 있다.

## 5. 평가 산식 (가장 중요)

```python
import numpy as np

TARGET_COLS = ["kpx_group_1", "kpx_group_2", "kpx_group_3"]
CAPACITY_KWH = {"kpx_group_1": 21600, "kpx_group_2": 21600, "kpx_group_3": 21000}

def metric(answer_df, pred_df):
    group_nmae, group_ficr = [], []
    for col in TARGET_COLS:
        actual = answer_df[col].to_numpy(dtype=float)
        forecast = pred_df[col].to_numpy(dtype=float)
        capacity = CAPACITY_KWH[col]
        valid = actual >= capacity * 0.10          # 이용률 10% 이상 시간대만 평가
        actual, forecast = actual[valid], forecast[valid]
        error_rate = np.abs(forecast - actual) / capacity
        group_nmae.append(np.mean(error_rate))
        unit_price = np.select([error_rate <= 0.06, error_rate <= 0.08], [4.0, 3.0], default=0.0)
        group_ficr.append(np.sum(actual * unit_price) / np.sum(actual * 4.0))
    one_minus_nmae = 1 - np.mean(group_nmae)
    ficr = np.mean(group_ficr)
    return 0.5 * one_minus_nmae + 0.5 * ficr, one_minus_nmae, ficr
```

이 산식에서 반드시 이해하고 최적화에 반영해야 할 특성:

1. **이용률 10% 미만 시간대는 아예 평가되지 않는다.** 저풍속 구간의 정밀도는 점수에 무관. 반대로 발전량이 큰 시간대를 틀리면 NMAE와 FICR 모두에서 손해가 크다 (FICR은 actual 가중이므로 고발전 시간대의 오차가 더 아프다).
2. **FICR은 계단식이다.** 시간별 오차율이 설비용량의 6% 이내면 만점, 6~8%면 3/4점, 8% 초과면 0점. 평균 오차를 줄이는 것과 별개로 "6% 밴드 안에 들어가는 시간의 비율"을 늘리는 것이 목표다. → MAE 최소화(중앙값 성향) 학습이 유리하며, 큰 오차 하나를 줄이는 것보다 경계(6~8%)에 걸친 시간대를 밴드 안으로 밀어 넣는 후처리가 점수에 직접적이다.
3. NMAE도 FICR도 분모가 설비용량이므로, 그룹 3(21,000kWh)은 그룹 1·2와 분모가 다르다. 그룹별로 따로 계산 후 단순 평균된다 — **세 그룹 모두 균형 있게 잘해야 한다.** 라벨이 적은 그룹 3이 병목이 될 가능성이 높다.
4. 후처리 기본기: 예측값을 `clip(0, capacity)` 범위로 자른다. 음수·용량 초과 예측 금지.

### 실험 로그 규칙

모든 실험은 `experiments/log.csv`에 아래 컬럼을 **모두** 기록한다. 정렬·비교의 1차 기준은 `total_score`다:

```
exp_id | date | git_hash | model/features 요약 | total_score | 1-NMAE | FICR | group별 NMAE(3개) | group별 FICR(3개) | val 기간
```

- `total_score`가 최종 순위 지표이므로 항상 첫 번째 지표.
- 단, 디버깅·개선 방향 판단에는 **NMAE를 먼저 본다**: NMAE는 연속 지표라 작은 개선도 안정적으로 잡히고, FICR은 계단식이라 노이즈가 크다. NMAE가 좋아졌는데 FICR이 정체되면 "오차 분포가 6% 경계 근처에 몰려 있다"는 신호이므로 오차율 히스토그램을 그려 확인한다.
- 그룹별 분해 지표를 반드시 남긴다 (그룹 3이 평균을 깎는지 즉시 확인 가능해야 함).

## 6. 피처 엔지니어링 우선순위 (기상 변수)

허브 높이가 117m이므로 **허브 높이에 가까운 바람이 핵심 입력**이다.

1순위 — 바람 (발전량 ∝ 풍속³):
- GFS: `heightAboveGround_100_100u/v` (100m, 허브에 가장 가까움), `heightAboveGround_80_u/v`, `heightAboveGround_10_10u/v`, `surface_0_gust`, 850hPa 바람(산악 상공 흐름)
- LDAPS: `heightAboveGround_10_10u/v`, 50m max/min 바람 4종, 경계층 바람(`XBLWS/YBLWS`)
- 파생: **풍속 `ws = sqrt(u²+v²)`**, 풍향 `wd = atan2`, `ws²`, `ws³`, 풍속의 연직 시어(100m ws − 10m ws), 격자간 풍속 분산(공간 불확실성), 풍향의 sin/cos 인코딩
- 10m/50m/80m/100m 풍속에서 **117m 허브 높이 풍속을 멱법칙/로그법칙으로 외삽**하는 피처 시도

2순위 — 공기 밀도 (출력 ∝ 밀도): 기온(`2_t`/`2_2t`), 기압(`surface_0_sp`), 습도 → `rho = p/(R·T)` 파생. 태백은 고지대라 밀도가 표준보다 낮다.

3순위 — 대기 안정도/난류: 경계층 높이(`etc_0_blh`), `planetaryBoundaryLayer_*`, 850/700/500hPa 온도·바람(상하층 시어), 시간대(주간 대류/야간 안정)

4순위 — 결빙·저감 조건: 기온+습도+강수(`prate`, `tp`, `lssrate`, 적설), 겨울철 블레이드 결빙으로 인한 저출력 탐지용

시간 피처: hour/month의 sin·cos, 계절. 격자 처리: 단지에서 가까운 격자 가중 평균 + 개별 격자 컬럼 병행 실험. **일사량·운량은 태양광용 변수이므로 우선순위 낮음** (안정도 프록시로만 보조 사용).

SCADA 활용: 터빈별 (풍속→출력) 파워커브를 학습해 "예보 풍속 → 예상 출력" 물리 피처를 만들고, ML 모델(GBDT 등)에 잔차 학습을 시키는 하이브리드 구조를 기본 베이스라인 다음 단계로 시도한다.

- 노트북 작성 규칙:
  - Phase별로 노트북을 분리한다: `01_preprocessing.ipynb`, `02_eda.ipynb`, `03_features.ipynb`, `04_model_selection.ipynb`, `05_tuning.ipynb`, 그리고 Phase 5는 대회 산출물 규정에 따라 `train.ipynb`(학습→모델 저장)와 `inference.ipynb`(모델 로딩→제출 생성)로 분리 (12번 규칙)
  - 각 셀은 한 가지 일만 하도록 작게 쪼개고, **셀 위에 마크다운 셀로 "무엇을 왜 하는지"를 먼저 적는다** (9번 문서화 규칙의 노트북 버전). 사용자가 위에서부터 셀을 하나씩 실행하며 따라올 수 있어야 한다
  - 노트북은 "Restart & Run All"로 처음부터 끝까지 오류 없이 재실행 가능해야 한다 (숨은 실행 순서 의존 금지)
  - 무거운 전처리 결과는 `data/processed/*.parquet`으로 저장하고, 다음 노트북은 그 캐시를 읽는 것으로 시작한다
- **노트북 우선, src 승격은 선택** — 새 처리 로직(전처리, 피처 등)은 반드시 **노트북에서 먼저 셀 단위로 작성·실행**해 사용자가 중간 결과(shape, head, 그림)를 확인하며 이해한 뒤 진행한다. **사용자 확인 없이 처음부터 .py 파이프라인으로 만드는 것을 금지한다.** 이후 재사용 방식은 두 가지 중 사용자가 택한다:
  - **(A, 기본) parquet 경유**: `01_preprocessing.ipynb`에서 train·test를 **동일한 셀 로직**으로 처리해 `data/processed/`에 각각 저장하고, 이후 노트북·`train`·`inference`는 그 parquet만 읽는다. 전처리 로직이 한 노트북에만 존재하므로 중복·불일치가 없다. 이 경우 `src/preprocess.py`는 만들 필요 없다.
  - **(B) src 승격**: 전처리·피처 로직을 `src/`의 함수로 정리해 `train`/`inference`가 각자 import한다. 원본에서 매번 처리하는 정석 파이프라인이 필요할 때 택한다. 승격은 노트북에서 로직이 확정되고 사용자가 승인한 뒤에만 한다.
  - 어느 쪽이든 **train과 inference가 완전히 동일한 전처리·피처 로직을 쓰도록** 보장하는 것이 목적이다 (12번 재현성 규칙). A는 "같은 셀로 둘 다 처리", B는 "같은 함수를 둘 다 import"로 이를 달성한다.
- **예외 — 다음 두 가지만 처음부터 .py로 작성한다** (틀리면 대회를 망치는 코드라 복사·변형을 막기 위함):
  - `src/metric.py`: 대회 산식 그대로, 수정 금지. 자체 지표를 추가로 만들어도 되지만 산식 자체는 한 글자도 건드리지 않는다
  - `src/submission.py`: 제출 파일 생성 + `validate_submission()`
  - 노트북에서는 `from src.metric import metric` 식으로 import만 해서 쓴다. 이 두 로직을 노트북 셀에 복사해 붙이지 않는다
- 랜덤 시드 고정, 실험은 재현 가능해야 한다.
- **주석 규칙 (필수)** — 이 프로젝트의 코드는 사용자가 읽고 학습하는 자료를 겸한다. 모든 코드는 아래 수준으로 주석을 단다:
  - 주석은 **한국어**로 작성한다
  - 모든 함수에 docstring을 단다: 무엇을 하는지, 입력(각 인자의 의미·형태·단위), 출력(형태·단위), 주의사항
  - "무엇을 하는지"만이 아니라 **"왜 이렇게 하는지"**를 적는다. 예: `# 풍속 세제곱 항 추가: 풍력 출력은 P = ½ρAv³로 풍속의 세제곱에 비례하기 때문`
  - 수식·물리 공식이 들어가는 곳은 공식 자체와 각 변수의 의미·단위를 주석으로 명시한다 (멱법칙 외삽, 공기 밀도, 파워커브 등)
  - pivot, merge, groupby 등 **데이터 형태가 바뀌는 지점마다 변형 전후의 shape/구조를 주석으로 표시**한다. 예: `# (시간×격자 long, 420,864행) → (시간당 1행 wide, 26,304행 × 격자별 컬럼)`
  - 매직 넘버 금지: `0.10`, `21600`, `117` 같은 숫자는 의미 있는 상수명으로 정의하고 출처를 주석으로 (예: `HUB_HEIGHT_M = 117  # info.xlsx, 전 터빈 공통 허브 높이`)
  - 문헌/EDA 근거로 만든 피처·처리에는 출처 명시: `# 근거: reports/phase1_eda.md §3` 또는 논문명
  - 노트북에서는 셀 위 마크다운(이 셀의 목적)과 코드 내 주석(이 줄의 이유)을 **둘 다** 단다. 마크다운이 있다고 주석을 생략하지 않는다
- 라벨 결측 처리: `kpx_group_3`의 2022년 빈칸은 0이 아니라 **결측**이다. 학습에서 제외하되, 그룹 1·2 데이터로 사전학습 후 그룹 3 파인튜닝 같은 전이 전략을 고려한다.

## 9. 작업 문서화 규칙 (필수)

**모든 작업은 마크다운 문서로 남긴다.** 코드만 만들고 끝내는 것을 금지한다. 목적은 (a) 사용자가 과정을 따라오며 학습할 수 있게 하고, (b) 발표 자료의 원천이 되게 하는 것이다.

**"대략" 쓰지 않는다 — 상세도 기준**: 문서만 읽고도 코드를 보지 않은 사람이 (1) 무슨 판단을 왜 내렸는지, (2) 데이터가 각 단계에서 어떻게 변했는지, (3) 결과 수치가 무엇을 뜻하는지를 재현·이해할 수 있어야 한다. "전처리를 했다", "피처를 만들었다" 같은 뭉뚱그린 문장은 금지. 동시에 **쉬운 말로 쓴다(8번 첫 규칙)** — 자세하게 쓰되 어렵게 쓰지 않는다. 전문용어는 풀어주고, 짧은 문장으로 나눈다. 아래 세부 규칙을 지킨다.

- 위치와 이름: `reports/phase{N}_{주제}.md` (예: `reports/phase0_preprocessing.md`, `reports/phase2_features.md`). 작업 단위가 크면 파일을 나눈다.
- 모든 문서는 아래 4개 구성을 반드시 포함한다:

```markdown
# (작업 제목)
## 1. 왜 (Why) — 이유와 근거
이 작업을 왜 하는가. 어떤 가설/문헌 근거/EDA 발견/대회 산식·규칙에서 출발했는가.
참고한 논문·자료가 있으면 제목과 링크(또는 출처)를 명시.
## 2. 어떻게 (How) — 과정 (여기를 특히 자세히)
- 처리 단계를 순서대로, **각 단계마다 별도 소제목이나 번호로** 기록한다. 한 단계 = 노트북의 셀 하나 또는 판단 하나에 대응.
- 각 단계에 다음을 적는다:
  - 무엇을 했는가 (사용한 데이터·컬럼·함수, 노트북 셀 번호/제목)
  - **데이터 형태 변화**: 입력 shape → 출력 shape, 행/열이 왜 그 숫자가 되는지 (예: "LDAPS 420,864행 = 26,304시각 × 16격자 → pivot 후 26,304행 × N열")
  - **내린 선택과 그 이유**: 결측·이상치·리샘플링·인코딩·집계 방식 등 판단마다 "무엇을, 왜 그렇게" (대안이 있었다면 왜 그걸 안 골랐는지)
  - 확인한 것: head()/통계/값 범위 등으로 검증한 내용과 그 결과
- **시도했다가 버린 방법과 버린 이유**를 반드시 남긴다 (같은 실수 반복 방지 + 발표 소명 자료).
```

## 13. 문헌 기반 도메인 지식 (피처·모델 선택의 출발점)

풍력 발전량 예측(day-ahead, NWP 기반) 문헌에서 반복적으로 확인되는 결론들. Claude Code는 이를 기본 가설로 삼고, 최신 논문 검색으로 보강한다:

- **출력은 풍속의 세제곱에 비례** (P = ½ρAv³·Cp). 풍속 자체보다 v², v³ 항이 트리 모델에서도 도움이 되는 경우가 많음.
- **허브 높이 풍속이 핵심**: 지상 10m 풍속만 쓰는 것보다 80~100m 풍속 또는 멱법칙(power law, α≈0.1~0.3)·로그법칙으로 허브고도(117m)로 외삽한 풍속이 예측력이 높음. 산악 지형은 α가 표준(0.143)과 다르므로 SCADA 실측으로 α를 추정하는 접근이 유효.
- **공기 밀도 보정**: ρ = p/(R_d·T)(습도 보정 포함 가능). 고지대·겨울철에 밀도 효과가 커서 동일 풍속에서도 출력 차이 발생.
- **파워커브 하이브리드가 강함**: NWP 풍속 → (SCADA로 학습한) 파워커브 → 물리 예측치를 만들고, ML이 잔차를 학습하는 물리+통계 하이브리드가 순수 ML보다 안정적이라는 보고가 다수.
- **풍향 효과**: 지형에 따른 wake/가속 효과로 같은 풍속이라도 풍향별 출력이 다름. 풍향은 sin/cos로 인코딩하고, 풍향×풍속 상호작용 검토.
- **NWP 앙상블/다중 소스**: LDAPS(고해상도, 국지 지형 반영)와 GFS(전지구, 상층 정보)를 함께 쓰면 단일 소스보다 개선된다는 것이 일반적 결론. 두 모델의 풍속 차이 자체가 불확실성 피처가 됨.
- **모델**: tabular NWP 피처 기반 단기 예측에서는 GBDT(LightGBM/XGBoost/CatBoost)가 강력한 기준선이며, 대회 우승 솔루션에서도 GBDT+피처엔지니어링+앙상블 조합이 지배적. 딥러닝은 대규모 데이터·다지점 공간 정보가 있을 때 우위.
- **결빙 손실**: 겨울철 기온 0°C 부근 + 고습도 + 강수 조건에서 출력 급감(icing). 태백은 고지대 한랭 지역이므로 결빙 플래그 피처와 겨울철 오차 분석이 중요.
