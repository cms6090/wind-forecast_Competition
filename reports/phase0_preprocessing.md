# Phase 0-2. 전처리 노트북(`01_preprocessing.ipynb`) — 로딩 → pivot → 조인 → 캐시 저장

## 1. 왜 (Why) — 이유와 근거

CLAUDE.md 11번 섹션의 Phase 0 작업 목록 중 "전처리는 `01_preprocessing.ipynb`에서 수행한다"
규칙에 따라, 원본 CSV를 다시 읽지 않고도 이후 모든 노트북(EDA, 피처, 모델)이 곧바로 쓸 수
있는 시간당 1행짜리 표를 만드는 것이 목적이다.

전처리가 필요한 근본적인 이유 두 가지 (CLAUDE.md 2번 규칙):
- 기상 예보(LDAPS, GFS)는 한 시각에 여러 격자가 행으로 쌓인 **긴 표(long format)**다.
  모델은 "시간 하나 = 행 하나"가 필요하므로 격자를 컬럼으로 펼치는 pivot이 필요하다.
- 기상 데이터와 발전량 라벨이 서로 다른 파일에 있어서, 시각을 기준으로 조인해야
  "이 시각 날씨 → 이 시각 발전량"이 한 행에 있는 학습용 표가 만들어진다.

추가로 SCADA(10분 단위 터빈 실측) 데이터의 단위를 라벨과 대조해 확정하는 것도 이
노트북에서 함께 했다. CLAUDE.md 2번 규칙이 "`power_kw10m` 단위 해석은 코드로 라벨과
크로스체크 후 확정"하라고 명시했기 때문이다.

## 2. 어떻게 (How) — 과정

작업은 `01_preprocessing.ipynb`에 30개 셀(마크다운 15 + 코드 15)로 나눠 기록했다.
아래는 각 단계의 판단과 결과다.

### 2-1. 경로 상수 설정
`DATA_DIR`, `TRAIN_DIR = data/train`, `TEST_DIR = data/test`, `PROCESSED_DIR = data/processed`를
상수로 정의했다. CLAUDE.md는 평평한 구조를 가정했지만 실제로는 `train/`, `test/` 하위
폴더가 있었으므로(Phase 0-1에서 이미 확인) 실제 구조에 맞췄다.

### 2-2. LDAPS train 로딩
`ldaps_train.csv`를 `parse_dates`로 읽고, 예상 행 수(420,864 = 26,304시각 x 16격자)와
실제 행 수가 같은지 `assert`로 확인했다. 통과.

### 2-3. LDAPS pivot 함수 작성 (`pivot_weather_wide`)
**무엇을 했나**: `forecast_kst_dtm`을 행으로, `grid_id`별 기상 변수를 `{prefix}_g{격자번호}_{변수명}`
컬럼으로 펼치는 함수를 만들었다. LDAPS/GFS 둘 다 재사용할 것이므로 `prefix` 인자로 구분했다
(둘 다 `surface_0_sp` 같은 이름이 겹치는 변수가 있어서, 접두어 없이 합치면 컬럼이 충돌한다).

**데이터 형태 변화**: 420,864행(26,304시각 x 16격자, long) → 26,304행 x 482열
(30변수 x 16격자 + `forecast_kst_dtm` + `ldaps_data_available_kst_dtm`, wide).

**확인한 것**: 같은 `forecast_kst_dtm`이면 16개 격자의 `data_available_kst_dtm`이 전부
같은지(하루 24시간 단위로 같은 예보가 공개되므로) `groupby().nunique()`로 확인 — 불일치 0건.

### 2-4. LDAPS test에 같은 함수 적용 — 실행 중 실제로 발견한 문제
train에서 만든 함수를 test에 그대로 적용했는데, **결측이 없어야 한다는 assert가 실패했다.**
원인을 추적한 결과, `ldaps_test.csv` 원본 자체에 정확히 3개 시각
(`2025-04-08 17:00`, `2025-06-18 18:00`, `2025-07-18 06:00`)에서 일부 기상 변수가
16개 격자 전부 비어 있었다 (총 752개 값, wide 기준). LDAPS train, GFS train, GFS test는
전부 결측 0건이었으므로 LDAPS 기상 자료가 이 3개 시각에만 일시적으로 끊긴 것으로 보인다.

이 3개 시각도 결국 예측·제출해야 하는 시각이라 행을 버릴 수 없다. 그래서
`pivot_weather_wide` 함수 안에 "시간순 정렬 후 선형보간(앞뒤 값으로 빈 값을 채움)"
처리를 추가했다. 결측이 없는 데이터(train, GFS test)에서는 이 처리가 아무 일도 하지
않으므로, train/test가 **완전히 같은 함수**를 타면서도 안전하다. 이 보간은 이미 공개된
같은 예보 자료의 앞뒤 값만 참고하는 것이라 발전량 라벨이나 미래 정보를 끌어오지 않으므로
CLAUDE.md 4번 누수 규칙과 무관하다.

**시도했다가 버린 방법**: 처음에는 결측 3개 시각을 통째로 제거하거나 0으로 채우는 방법을
생각했지만, (a) test는 8,760행을 모두 채워 제출해야 해서 제거가 불가능했고, (b) 0으로
채우면 물리적으로 말이 안 되는 값(예: 기압이 0)이 들어가 오히려 위험했다. 선형보간이
가장 안전한 선택이었다.

### 2-5. GFS train/test pivot
같은 `pivot_weather_wide(prefix="gfs")` 함수를 재사용했다. GFS는 결측이 전혀 없었다.
- GFS train: 236,736행(26,304시각 x 9격자) → 26,304행 x 317열(35변수 x 9격자 + 2)
- GFS test: 78,840행(8,760시각 x 9격자) → 8,760행 x 317열

### 2-6. 라벨(`train_labels.csv`) 로딩 및 결측 확인
26,304행, 예상과 일치. 결측 확인 결과:
- `kpx_group_1`: 104개 결측, `kpx_group_2`: 103개 결측 (산발적)
- `kpx_group_3`: 총 8,766개 결측. 이 중 8,759개는 2022년 전체 시간(2022년에 해당하는
  라벨 행 수 자체가 8,759개 — 데이터가 2022-01-01 01:00부터 시작해 2022-12-31 23:00에서
  끝나므로 8,760이 아니라 8,759임)과 정확히 일치했고, 나머지 7개는 2023~2024년 중
  산발적으로 빠진 값이었다. CLAUDE.md 8번 규칙대로 이 결측은 0이 아니라 그대로 NaN으로
  남겨뒀다(학습 단계에서 처리).

### 2-7. SCADA 단위 검증 (가설 A vs 가설 B)
**두 가설**: (A) `power_kw10m`이 "10분간 평균 출력(kW)"이라 1시간 발전량은 6개 값의
평균, (B) 이름과 달리 이미 "10분간 발전량(kWh)"이라 1시간 발전량은 6개 값의 합.

**검증 방법**: 2022-01-01 02:00~08:00 7개 시간에 대해 그룹1(VESTAS 1~6호기 합)의
가설A/가설B 추정치를 실제 라벨과 비교했다. 결과:

| 시각 | 가설A(평균) 오차율 | 가설B(합계) 오차율 |
|---|---:|---:|
| 02:00 | 83.1% | 1.5% |
| 03:00 | 83.2% | 0.8% |
| 04:00 | 83.0% | 1.9% |
| 05:00 | 83.1% | 1.3% |
| 06:00 | 83.1% | 1.2% |
| 07:00 | 83.3% | 0.2% |
| 08:00 | 83.4% | 0.4% |

가설B(합계)의 오차율이 2% 이내로 압도적으로 낮았다. **결론: `power_kw10m`은 이미
kWh 스케일의 10분 에너지값이고, 1시간 발전량 = 6개 구간의 합(sum)이다.** CLAUDE.md
2번 규칙이 제안했던 "합산 후 /6"(=평균, 가설A)은 틀린 방향이었고, 실제 코드 대조로
"나누지 않는 합"이 맞다는 것을 확정했다. 이 노트북 안에 검증 과정과 근거를 그대로
남겨서, 왜 CLAUDE.md 초안 제안과 다른 방식을 택했는지 소명할 수 있게 했다.

### 2-8. SCADA 시간 집계 함수 (`aggregate_scada_hourly`)
확정된 "합" 방식으로 전체 기간을 집계했다. 터빈별로 `{터빈}_power_kwh`(합),
`{터빈}_ws_mean`/`{터빈}_wd_mean`(평균), 그리고 그 시간에 실제로 있었던 10분 샘플
개수(`n_samples`, 정상은 6)를 만들었다.
- VESTAS: 26,304행 x 38열. `n_samples==6` 비율 99.996% (파일 시작 경계의 첫 시간 1개만
  샘플 1개로 불완전 — 자연스러운 경계 현상).
- UNISON: 17,544행(2023~2024년, 2년치) x 17열. `n_samples==6` 비율 100%.

### 2-9. 그룹 합계 vs 라벨 전체 기간 대조
| 그룹 | 대조 시간 수 | 비율 평균(라벨/SCADA) | 비율 표준편차 |
|---|---:|---:|---:|
| kpx_group_1 | 22,412 | 1.030 | 0.926 |
| kpx_group_2 | 22,423 | 1.031 | 1.083 |
| kpx_group_3 | 14,432 | 1.074 | 1.103 |

평균 비율은 세 그룹 모두 1.03~1.07로 "합" 방식이 맞다는 것을 다시 확인해준다. 다만
**표준편차가 매우 크게 나왔는데, 이는 SCADA 합계가 0에 가까운 저풍속 시간대에서
아주 작은 절대오차도 비율로 보면 크게 부풀려지기 때문**이다(예: SCADA 합계가 1kWh인데
라벨이 5kWh면 비율이 5.0). 이 표는 "합 방식이 맞는 방향이다"는 근거로는 유효하지만,
그룹별 SCADA 신뢰도를 비교하는 지표로 그대로 쓰기엔 오해의 소지가 있다. Phase 1
EDA에서는 이 비율 비교를 대회 산식처럼 "설비용량 10% 이상 시간대만" 필터링해서 다시
계산하는 것이 더 정확할 것이다 (다음 단계로 남겨둠).

### 2-10. 학습 데이터 병합
`ldaps_train_wide`, `gfs_train_wide`, `labels`(컬럼명 `kst_dtm`→`forecast_kst_dtm`으로
통일)를 `forecast_kst_dtm` 기준 inner join했다. 세 표 모두 26,304행이므로 조인 후에도
26,304행이 유지되는지 assert로 확인(통과). LDAPS와 GFS의 `data_available_kst_dtm`이
서로 일치하는지도 확인(둘 다 "전날 13:00 공개" 규칙을 따르므로 일치해야 정상 — 실제로
100% 일치), 일치하는 값 하나만 `data_available_kst_dtm` 컬럼으로 남기고 중복 컬럼은
제거했다. 기상 피처 컬럼에는 결측이 0건인지 재확인(통과, 라벨 컬럼의 결측은 정상이므로
검사 대상에서 제외). **최종 shape: 26,304행 x 800열.**

### 2-11. 평가용(test) 데이터 병합
같은 방식으로 `ldaps_test_wide` + `gfs_test_wide`를 병합했다(라벨 없음). train과 test의
기상 피처 컬럼 집합이 완전히 같은지(`set` 비교)도 확인했다 — 다르면 이후 모델 입력이
어긋나기 때문이다. **최종 shape: 8,760행 x 797열** (train과의 차이 3 = `kpx_group_1/2/3`).

### 2-12. parquet 캐시 저장
- `data/processed/train_merged.parquet` (127.4 MB)
- `data/processed/test_merged.parquet` (43.0 MB)
- `data/processed/scada_vestas_hourly.parquet`, `data/processed/scada_unison_hourly.parquet`

### 2-13. 재현성 확인
저장한 parquet을 다시 읽어 shape·기간·컬럼 수가 예상과 같은지 재확인했다(통과). 노트북
전체를 nbclient로 "Restart & Run All"과 동일하게 처음부터 끝까지 실행해, 오류 없이
재현되는 것을 확인했다.

## 3. 결과 (Result)

| 산출물 | shape | 비고 |
|---|---|---|
| `data/processed/train_merged.parquet` | 26,304 x 800 | LDAPS(480) + GFS(315) + 라벨(3) + 시각/공개시각(2) |
| `data/processed/test_merged.parquet` | 8,760 x 797 | LDAPS(480) + GFS(315) + 시각/공개시각(2) |
| `data/processed/scada_vestas_hourly.parquet` | 26,304 x 38 | 터빈 12기 x (kWh, ws, wd) + n_samples |
| `data/processed/scada_unison_hourly.parquet` | 17,544 x 17 | 터빈 5기 x (kWh, ws, wd) + n_samples |

핵심 결론: **SCADA `power_kw10m`은 "합(sum)"으로 시간 집계해야 라벨과 일치한다** (오차 2%
이내, CLAUDE.md 초안의 "합산 후 /6" 제안과 다름 — 코드 대조로 확정).

LDAPS test 원본에 3개 시각(752개 값, 전체의 0.03% 수준)의 결측이 있었고, 시간 선형보간으로
메웠다. 그 외 LDAPS train·GFS train·GFS test는 결측 0건.

## 4. 해석과 다음 단계 (So what)

- 이제부터 `02_eda.ipynb`부터는 원본 CSV를 다시 읽지 않고 `data/processed/*.parquet`만
  읽으면 된다. 로딩 속도가 크게 빨라진다(원본 CSV 합계 약 250MB → parquet 170MB, 게다가
  pivot·조인이 이미 끝난 상태).
- SCADA "합" 방식 확정은 Phase 2(피처 생성)의 파워커브 추정, 예보 풍속 보정 모델 학습에
  바로 쓸 수 있는 기반이 됐다.
- 그룹3(UNISON)의 라벨 대비 SCADA 비율 표준편차가 가장 크게 나온 것은(1.103)
  CLAUDE.md가 예상한 대로 "그룹3이 병목이 될 가능성"과 방향이 일치한다. 다만 위에서
  설명했듯 이 표준편차 수치 자체는 저풍속 구간의 비율 왜곡 때문에 부풀려져 있어서,
  Phase 1 EDA에서 설비용량 10% 이상 구간만 필터링해 다시 확인할 필요가 있다.
- `train_merged.parquet`(127MB)와 `test_merged.parquet`(43MB)는 용량이 크지만
  `.gitignore`의 `data/` 규칙에 이미 포함되어 자동으로 git 추적에서 제외된다 — 별도
  조치 불필요.
- 다음 단계: `02_eda.ipynb`에서 그룹별 발전량 패턴, 파워커브(SCADA 풍속-출력 산점도),
  예보-실측 풍속 비교, 이용률 10% 이상 시간대 비율 등을 확인한다 (CLAUDE.md 11번 Phase 1).
