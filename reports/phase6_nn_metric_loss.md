### 1-2. 실패의 공통 원인 — 한 문장으로

**전부 "예보 피처의 결정론적 함수"를 추가하는 시도였다.**

GBDT가 이미 그 피처들을 갖고 있다면, 그것들로 만든 어떤 새 컬럼도 **정보를 늘리지 않는다.**
트리는 필요하면 스스로 그 조합을 만든다. 새 컬럼은 `feature_fraction=0.7` 아래서
좋은 피처를 후보에서 밀어낼 뿐이다.

특히 **"우리 피처 엔지니어링이 공간 정보를 버렸다"는 가설**(HREFTC 2024 우승팀이 raw 격자점을 썼다는 근거)은
직접 재서 기각했다. **원시 800컬럼만 쓰면 −0.0116으로 유의하게 나쁘다.**
우리의 179개 피처는 원시값보다 확실히 낫다.

### 1-3. 결정적 관측 — 곡선 위에는 답이 없다

| 설정 | 1−NMAE | FICR |
|---|---:|---:|
| Phase 3 L1 (2024 홀드아웃) | 0.8688 | 0.3459 |
| A 고정 τ (2024 홀드아웃) | 0.8636 | 0.3979 |
| B 결정이론 (2024 홀드아웃) | 0.8653 | 0.3977 |

**1등은 두 축 모두에서 앞선다.** τ를 올리든 내리든, A를 B로 바꾸든,
우리는 **같은 트레이드오프 곡선 위를 미끄러질 뿐**이었다.

곡선을 바깥으로 미는 방법은 하나뿐이다: **같은 정보에서 더 좁은 조건부 분포를 만드는 것.**
정보를 늘리는 시도가 모두 실패했으므로, 남은 것은 **목표(손실함수)를 바꾸는 것**이었다.

## 2. 어떻게 (How) — 산식을 손실함수로

### 2-1. 계단 함수를 미분 가능하게

대회 점수(그룹 하나, 이용률 단위 `u = 1`):

$$ \text{score} = \frac12\Big(1 - \overline{|\hat y - y|}\Big) + \frac12 \cdot \frac{\sum_i y_i\, p(e_i)}{4\sum_i y_i},
\qquad e_i = |\hat y_i - y_i| $$

`p(e)`는 `e ≤ 0.06`에서 4, `e ≤ 0.08`에서 3, 그 밖에서 0인 **계단 함수**다. 미분이 안 된다.
시그모이드 두 개로 정확히 그 모양을 만든다.

$$ p_{\text{soft}}(e) = 3\,\sigma\!\left(\frac{0.08 - e}{T}\right) + \sigma\!\left(\frac{0.06 - e}{T}\right) $$

| 구간 | 첫째 항 | 둘째 항 | 합 | 실제 단가 |
|---|---:|---:|---:|---:|
| `e ≪ 0.06` | 3 | 1 | **4** | 4 ✔ |
| `0.06 < e < 0.08` | 3 | 0 | **3** | 3 ✔ |
| `e > 0.08` | 0 | 0 | **0** | 0 ✔ |

손실은 `-score`다. 구현은 `src/nn.py`의 `metric_loss()`.

**이 손실이 하는 일**: 어떤 샘플이 6% 밴드에 들어갈 가망이 있으면 그쪽으로 강하게 당기고,
가망이 없으면 그냥 L1처럼 다룬다. **시각마다 다르게 행동한다.**
분위수 τ를 올려 **모든 시각을 똑같이** 위로 미는 방식과 근본적으로 다르다.

`T`(계단을 부드럽게 하는 폭)는 작을수록 실제 계단에 가깝지만 기울기가 날카로워진다. 교차검증으로 골랐다.

### 2-2. LightGBM으로 먼저 시도했다 — 그리고 실패한 이유를 규명했다

**처음 생각**: LightGBM은 커스텀 목적함수(gradient + hessian)를 받는다.
신경망 없이도 산식을 직접 최적화할 수 있어야 한다.

**결과**: CV가 0.6059 → 0.5641로 **무너졌다.**

원인을 가르기 위해 **밴드 항을 완전히 끄고(k=0) 순수 매끄러운 L1만** 돌려 봤다.
잘 나오면 기계장치는 멀쩡하고 밴드 항이 문제, 무너지면 기계장치가 고장이다.

| 설정 | CV | 평균 트리 |
|---|---:|---:|
| **LightGBM 내장 `l1` (채점행 학습)** | **0.5897** | — |
| 커스텀 k=0 (밴드 끔), δ=0.005 | 0.4799 | **1그루** |
| 커스텀 k=0, δ=0.02 | 0.5056 | 69 |
| 커스텀 k=0, δ=0.05 | 0.5474 | 79 |
| 커스텀 k=0, δ=0.10 | 0.5814 | 99 |
| 커스텀 밴드 ON, δ=0.10 | 0.5766 | 379 |

**밴드 항을 완전히 끄고 순수 L1만 돌려도 내장 `l1`보다 못하다.**

**원인**: 부스팅은 잎 값을 `−Σg / Σh`로 정한다. LightGBM 내장 L1은 여기에 더해
**잎 값을 잔차의 중앙값으로 다시 계산하는 특수 보정**(`RenewTreeOutput`)을 한다.
커스텀 목적함수에는 그 보정이 없다.
게다가 `δ`(매끄러운 |r|의 폭)가 작으면 `r=0`에서 헤시안이 200, `|r|=0.1`에서 0.025로
**8,000배 차이**가 나 잎 값이 폭주한다. δ를 키워 조건수를 고치면(0.10) 이미 L1이 아니라 L2에 가깝다.

그리고 잘 조건화된 δ=0.10에서 **밴드 항을 켜면 −0.005**다.
비볼록한 밴드 보너스가 만든 기울기를 부스팅의 잎 값 규칙이 제대로 처리하지 못한다.

> **결론**: LightGBM 안에서 산식 직접 최적화는 안 된다. **경사하강법에는 그런 제약이 없다.**

> **덤으로 잡은 버그**: 처음 구현에서 CV가 0.44까지 떨어졌다. 원인은
> `feval`에 넘어오는 `preds`가 이미 `init_score`를 포함하는데 거기에 `init`을 또 더한 것이었다.
> LightGBM 규약: **커스텀 목적함수와 `feval`의 `preds`는 `init_score` 포함, `booster.predict()`는 미포함.**
> 헷갈리기 쉬운 지점이라 `src/nn.py`와 노트북 주석에 명시했다.

### 2-3. 신경망 설계

`src/nn.py`에 두고 `train.ipynb`/`inference.ipynb`가 **import**한다
(CLAUDE.md 12-3: "공통 로직은 src/의 함수로 두고 양쪽에서 import").

| 항목 | 값 | 이유 |
|---|---|---|
| 구조 | 179 → 256 → 256 → 1, GELU, BatchNorm, Dropout 0.15 | 표본 17k에 맞는 작은 모델 |
| 출력 | **sigmoid** | 발전량은 [0, 설비용량] 밖이 물리적으로 불가능. 애초에 못 벗어나게 |
| 학습 행 | **채점 대상 행만** (이용률 ≥ 10%) | 산식의 정의역이 그곳이다 |
| 배치 | **전체 배치(full-batch)** | FICR 항이 "전체 합의 비율"이라 미니배치면 추정이 흔들린다 |
| 최적화 | AdamW (lr 1e-3, wd 1e-4) + 코사인 스케줄 + 기울기 클리핑 | |
| 조기 종료 | **내부검증의 대회 산식** (MAE 아님) | 산식을 최적화하니 산식으로 멈춰야 한다 |
| 장치 | **CPU 전용** | GPU 비결정성 회피 (CLAUDE.md 12-3 재현성) |

**표준화(mu, sd)는 학습 데이터에서만 fit**하고 `scaler_{group}.npz`에 저장한다 (CLAUDE.md 4번).

### 3-4. 2024 홀드아웃 — 정직한 확인

### 3-6. 왜 블렌드가 두 축을 모두 개선하는가

Phase 3에서 이렇게 기록했다: *"단순 평균 앙상블은 FICR을 깎는다"* (LightGBM+XGBoost+CatBoost, 0.6074 → 0.6051).
지금은 정반대다. **모순이 아니다.**

- **그때**: **같은 귀납 편향**(축 정렬 분할)을 가진 트리 셋. 평균이 예측을 "평균값" 쪽으로 끌어당겼다.
  우리 산식은 중앙값/최빈값 성향을 원하므로 FICR이 깎였다.
- **지금**: **축 정렬 분할(트리)** vs **매끄러운 함수(신경망)**. 근본적으로 다른 오차를 내므로 상쇄된다.

### 3-7. Phase 3의 진단은 옳았고, 처방이 틀렸다

`phase3_model_selection.md` §3-4에 이렇게 적었다.

> 매끄러운 함수 계열(Ridge)이 NMAE는 따라잡지만(0.8688 vs 0.8690) FICR에서 0.03 뒤진다.
> 신경망도 같은 계열이므로, 이 계단형 구조를 배우려면 **훨씬 많은 데이터가 필요하다.**

**진단은 정확했다.** 매끄러운 함수는 NMAE는 잡고 FICR을 못 잡는다.
실제로 MLP + L1은 CV 0.5926(FICR 0.3271)으로 그 예측 그대로였다.

**그런데 결론이 틀렸다.** 필요한 것은 **더 많은 데이터**가 아니라 **FICR을 목표로 삼는 손실함수**였다.
산식 손실을 붙이자 매끄러운 함수의 약점(FICR)이 사라지고 강점(NMAE)은 남았다.

> **교훈**: "모델 계열 A는 지표 X에 약하다"를 관찰했을 때,
> 그것이 **표현력의 한계**인지 **목표 설정의 문제**인지 구별해야 한다. 우리는 전자로 단정했다.

### 3-8. 재현성 (CLAUDE.md 12-3)

|---|---|---|---:|
| kpx_group_1 | True | True | 0.000e+00 |
| kpx_group_2 | True | True | 0.000e+00 |
| kpx_group_3 | True | True | 0.000e+00 |

**비트 단위로 동일하다.** LightGBM은 `deterministic=True` + 스레드 고정,
PyTorch는 `use_deterministic_algorithms(True)` + CPU + 시드 고정.

### 3-9. 제출 (exp017)

## 4. 해석과 다음 단계 (So what)

### 4-1. 무엇이 달랐는가

열두 번의 실패는 전부 **"정보를 늘리려는 시도"** 였다. 성공한 하나는 **"목표를 바꾼 것"** 이다.

| | 바꾼 것 | 결과 |
|---|---|---|
| 실패 12건 | 입력 (피처, 격자, 앙상블) | 트리가 이미 만들 수 있는 것 → 정보 0 |
| **성공 1건** | **손실함수 (목표)** | 같은 입력에서 다른 예측 → **곡선이 바깥으로** |

### 4-2. 아직 안 해본 것 (신경망만 할 수 있는 것)

**SCADA를 보조 감독신호로 쓰는 멀티태스크 학습.**
우리는 SCADA를 파워커브와 선형 보정식에만 썼다. 그런데 SCADA에는
**터빈 17기 × 10분 단위 × 3년치**의 출력·풍속·풍향이 있다.

신경망이라면 "예보 → 발전량"을 배우면서 동시에
- "예보 → 각 터빈의 나셀 풍속"
- "예보 → 각 터빈 출력"

을 **보조 과제(auxiliary head)** 로 학습시킬 수 있다. 표현 학습이 풍부해진다.
**GBDT로는 구조적으로 불가능한 일이고, 우리가 가진 정보 중 아직 안 쓴 마지막 큰 덩어리다.**

누수 검토: SCADA는 학습 기간에만 존재하고, **보조 타깃으로만** 쓴다(추론 시 입력이 아니다).
CLAUDE.md 4번의 허용 용법 (a) "학습 시 보조 타깃/사전학습"에 정확히 해당한다.
